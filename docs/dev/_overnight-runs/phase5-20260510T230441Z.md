### Root cause of the terminal block

A pre-existing /16 docker network owned by a sibling worktree
(`lsierant_devcontainer-test`) occupies the whole 172.16.0.0/16
space. The new /23-stack allocator (commit 16244652b "drop legacy /16
scheme") allocates slots in /23 increments within 172.16.0.0/12 but
**doesn't detect or migrate around pre-existing /16 networks created
by the old scheme**. `wt-ctl network prune --networks --dry-run`
correctly identifies the network as a candidate but refuses to
remove it because the owning worktree still exists.

This is not strictly a Phase-1 regression — it's a migration gap from
the legacy /16 scheme. It's listed in the UX section below.

The pre-authorization in the orchestration prompt scopes destructive
actions to "this worktree, its EVG host, kube-forwarding-proxy" — I
am not authorized to remove a sibling worktree's docker network, and
the action was correctly denied by the harness.

## Section C — Failure isolation

### Phase-1 regressions fixed on `lsierant/devcontainer`

1. **`69f79d701`** — `wt-ctl: evg spawn — parse plain-text 'keys list', prefer evg-host`. The `_resolve_key_name` helper called `evergreen keys list --json`, but the upstream CLI has no `--json` flag on that subcommand — every fresh spawn raised "cannot resolve a public-key name". Fix: parse the plain-text `Name: 'NAME', Key: 'TYPE BLOB COMMENT'` format; prefer `evg-host` (matches `~/.ssh/evg-host` per the project's SSH policy memory) when registered; env var `MCK_DEVC_EVG_KEY_NAME` still wins. **6 new unit tests** covering all three resolution branches plus end-to-end "spawn passes `--key evg-host`".
2. **`61024b669`** — `wt-ctl: evg spawn — resolve placeholder via pre-create snapshot diff`. `evergreen host create` returns only an `evg-*` placeholder id (the real `i-*` is AWS-assigned asynchronously); the poll loop's id-lookup and display-name fallback both fail until tagging happens, and tagging requires the `i-*` id. With a sibling spawn racing, the "freshest untagged" fallback can't pick the right host. Fix: snapshot all `--mine` host ids immediately before `host create`; use set-diff in the poll loop to identify the newly-spawned host (only when exactly one new candidate; keep polling otherwise). **1 new unit test** for the concurrent-sibling scenario.
3. **`51cd6f3b2`** — `wt-ctl: net_allocate — invalidate input_hash when .env is partial`. Counterpart to sibling-agent's `10b001a44`. The repair branch added by `10b001a44` runs inside `_do_net_allocate`, but the orchestrator's resume short-circuits the phase when `status=ok` and `input_hash` matches — so the repair never fires on a partially-written .env. Fix: include the set of missing derived keys in the hash, so an incomplete .env mismatches the recorded hash and forces the phase to re-run.

### UX issues found

- `wt-ctl create --help` advertises a required positional `branch`, but the orchestration prompt template says "no `--branch` needed". Make either the positional optional (defaults to current branch) or update the prompt.
- `wt-ctl kfp --help` lists `status / start / stop / register` subcommands but only documents `register`; the other three appear with no help text.
- `wt-ctl attach --help` doesn't describe the positional `args`.
- Parallel phases `evg_prepare + dc_build` don't fail-fast on terminal errors. When `evg_prepare` raises immediately with an unrecoverable config error (missing key, missing tag), `dc_build` continues for its full ~5 min image build before the orchestrator merges results and surfaces the failure. Could short-circuit on a `WtCtlError` from one thread while the other is still running. (Violates `feedback_fail_fast_no_masking.md` for parallel phases.)
- `wt-ctl status` renders `last_create  failed at dc_build` (the rightmost phase that's not `ok`) when the actual failure was earlier in `evg_prepare`. Misleading because state.json still shows `dc_build` as `in_progress` (it was killed mid-flight, not failed).
- `evg_prepare`'s idempotency: the phase records `ok` only after the whole script completes. If the host has been spawned + kind has been recreated but the orchestrator is killed during the kubeconfig copy, `--resume` re-runs the whole script — including the full kind teardown + recreate (~7 min). Add sub-checkpoints (host spawned → ok, kind recreated → ok, kubeconfig copied → ok) so resume picks up at the last completed sub-step.
- Legacy `/16` docker networks block all `/23` allocations within 172.16.0.0/12. `wt-ctl network prune --networks` correctly identifies these as candidates but skips them when an owning worktree exists. Add detection at net_allocate time: if any /16 network at 172.X.0.0 exists where X∈[16,31], surface a clear actionable error ("legacy /16 network NAME blocks /23 allocation; run `docker network rm NAME` after stopping its compose project").
- Error hint pointed at `evergreen keys list --json` which isn't a valid flag (fixed in 69f79d701).
- `evergreen host modify` rejects `--tag name=...` and the `--name` flag without a companion `--tag/--expire/...` flag; this was already handled by the sibling agent's `f84bacfbd` (using `wt-ctl=<name>` as the side-label tag).
- Nohup-detached orchestrator runs (the pattern this overnight uses) leave state.json's phase records as `in_progress` when SIGTERM'd mid-flight. There's no automatic GC of dangling `in_progress` records on a fresh `--resume`, so the next run reports `last_create failed at dc_build` (rendering bug) but actually proceeds normally.
