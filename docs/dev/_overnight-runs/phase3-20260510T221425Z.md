## UX gaps found (follow-ups, not Phase 3's job)

These belong in Phase 1 (extending wt-ctl) or Phases 4–5; surfacing them here so the next cleanup can pick them up. The skill text deliberately did **not** paper over these gaps by inventing descriptions — every entry below appears in the verb table only with the verbatim flag, no documentation beyond what's in the source.

| # | Severity | Verb / flag | Gap |
|---|---|---|---|
| 1 | high | `create --multi-cluster`, `--skip-recreate`, `--skip-evg`, `--skip-devcontainer`, `--skip-prepare-e2e`, `--force`, `--resume`, `--restart-from`, `--evg-host-name` | Missing `help=` text. Users have to read `cli.py` (or the plan doc) to know that `--restart-from` takes a phase name from a fixed list, that `--skip-recreate` controls the kind-recreate inside `prepare-local-e2e`, that `--resume` only skips phases whose `input_hash` matches, etc. |
| 2 | high | `delete --delete-branch`, `--keep-evg`, `--keep-worktree`, `--keep-stack`, `--keep-om-projects`, `--evg-host-name` | Same — no `help=`. Users can't tell from `--help` what each `--keep-*` flag preserves. |
| 3 | high | `status [branch] [--all]` | Positional `branch` and `--all` both undocumented. The `branch` argument is actually unused by `cmd_status_all` (verified by reading `cli.py`); it's vestigial — should either be wired up or removed. |
| 4 | medium | `network release branch` / `network allocate branch` | The positional is named `branch` but the registry stores `branch_dir` (slash-replaced). Mismatch will trip users who paste `lsierant/foo` and find no entry; the actual key is `lsierant_foo`. Either rename the positional or document the transform. |
| 5 | medium | `evg extend --hours` | No help text and no indication the default is 4 (taken from `cli.py`). Users guessing from `evg_host.sh` may expect 8. |
| 6 | medium | `evg terminate --host-id`, `evg extend --host-id` | No help text. Falls back to "infer from current worktree" only if omitted — that behaviour isn't visible from `--help`. |
| 7 | medium | `logs [phase] [-f]` | Phase name space is hardcoded in `paths.logs_dir` / `cmd_logs` (build, evg, up, refresh, prepare-e2e, create, delete, and a legacy `dc_*` fallback). `--help` should enumerate them, ideally as `choices=`. |
| 8 | low | `build --no-cache`, `network prune --dry-run`, `network release --dry-run`, `ctx switch ctx`, `wt new -f`, `wt rm -f` | All missing `help=`. Self-evident enough to be tolerable but inconsistent with the global `--quiet` / `--color` which *do* have help text. |
| 9 | low | `attach args (REMAINDER)` | `--help` shows `...` with no positional doc. Not obvious from `--help` that bare `attach` runs tmuxp and any args exec verbatim with `MCK_NO_TMUX=1`. |
| 10 | low | `network prune --networks` | Help mentions hard-coded range `172.[16-31].x` — implicit dependency on task #16 (prefix expansion to 255). When that lands, this string needs updating. |
| 11 | low | Nested subparsers (`network`, `evg`, `kfp`, `om`, `ctx`, `wt`) | Top-level `wt-ctl <verb> --help` for these only lists sub-names without describing them. Adding `help="…"` to each `sn_sub.add_parser(...)` call would surface much more in `--help` and dramatically reduce the need to read the source. |
| 12 | cosmetic | Variable shadow in `cli.py` | Two `sc =` variables — one for `create` (`sc = sub.add_parser("create", …)`), one for `ctx` (`sc = sub.add_parser("ctx", …)`). Not a runtime issue but a readability hazard during edits. |

None of these change the user-facing flow today; they're all about discoverability via `--help`. The skill leaves them out so as not to invent contracts. If/when they're fixed, the verb table in `local-kind-dev/SKILL.md` is the natural place to mirror the new help text.
