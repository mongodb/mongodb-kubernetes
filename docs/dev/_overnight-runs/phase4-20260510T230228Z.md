## Section C — Failures categorised

### Phase-1 regression — `evg spawn` rejects reserved tag key (fixed)

`scripts/dev/wt_ctl/domains/evg.py` issued the post-create rename via
`evergreen host modify --host <id> --name <name> --tag name=<name>`.
The EVG API rejects the `name` tag key with HTTP 400 ("tag 'name'
cannot be modified"), so the whole modify call returned `rc=1`, the
rename silently failed (warn-only path), and the host stayed
display-nameless. Subsequent resume probes (`_find_live_by_display_name`)
then returned `None` and re-spawned a fresh host every time.

Fix: use a side-label tag `wt-ctl=<name>` (reserved key avoided);
the rename portion still rides the same call. Test
`test_tag_and_rename_after_aws_id` updated.

Commit: `f84bacfbd wt-ctl: evg spawn — drop reserved tag-key,
surface keys-list diag` on `lsierant/devcontainer`.

### Phase-1 regression — `_resolve_key_name` swallowed diagnostics

When `evergreen keys list` returned a transient non-zero or an
unparseable stdout (observed once during the very first parallel
`evg_prepare + dc_build` race), the spawn raised
`"cannot resolve a public-key name. Set MCK_DEVC_EVG_KEY_NAME or pass
--key"` with no rc / stderr context. The user's `evg-host` key was
clearly registered, so the message was misleading.

Fix: `_resolve_key_name` now returns `(name, diag)`; the empty-name
branch surfaces rc + stderr-tail (or stdout-head for parse failures)
so the next transient is diagnosable from the log alone. Two tests
updated to inspect the new tuple.

Same commit as above.

### Phase-1 regression — `net_allocate` skips on partial .env (fixed)

`_do_net_allocate` short-circuited as soon as it saw
`MCK_DEVC_NET_PREFIX=<N>` in `.devcontainer/.env`, even when the four
derived vars (`MCK_DEVC_NET_X`, `MCK_DEVC_NET_Y_BASE`,
`MCK_DEVC_NET_Y_VIP`, `MCK_DEVC_PROXY_PORT`) had never been written.
With only the prefix line present, `compose.yml`'s inline default
`${VAR:-default}` fell back to `X=16, Y_BASE=0` (stack index 0), so
every parallel worktree tried to attach to `172.16.0.0/23` — instant
"networks have overlapping IPv4" 400 from the Docker daemon.

Fix: detect missing derived keys, compute `stack_params(prefix)`,
append only the missing lines. New unit test
`test_net_allocate_repairs_partial_env`.

Commit: `10b001a44 wt-ctl: net_allocate — repair partial
.devcontainer/.env on resume` on `lsierant/devcontainer`.

### Pre-Phase-1 design gap — `/16` zombies invalidate `/23` allocations

A long-lived docker network on this host
(`lsierant_devcontainer-test_devcontainer_devcontainer`) claims
`172.16.0.0/16` (legacy pre-`/23` provisioning). The registry's
allocator computes `blocked_xs` from observed docker networks by
X-octet, so X=16 is correctly marked blocked — but only if the
registry **calls** allocate. Our partial `.env` (prefix=29 → X=16)
predated the call and pinned a /23 inside the zombie /16. Even after
the partial-env repair, `docker network create` still refuses to add
a /23 inside a pre-existing /16 — the new /23 is geometrically
inside the old /16, regardless of the new allocator's awareness.

Workaround applied: `rm .devcontainer/.env`, then
`scripts/dev/wt-ctl network allocate` picked `prefix=1280 → X=26,
Y_BASE=0` (a free /23 well outside the zombie /16). compose up
succeeded on the first try after this; containers carrying the old
`INTERFACE=172.29.0.10` env had to be `docker compose down`'d and
recreated to pick up the new X=26.

Recommendation (not implemented — needs broader buy-in):

- `wt-ctl network` should grow a "scan for /16-or-larger zombies in
  172.16.0.0/12" check and refuse to allocate inside any such range
  (currently `blocked_xs` only excludes the literal X used by the
  zombie, not the supernet it covers).
- The migration story for pre-Phase-1 .env files (only prefix line)
  should be documented; `wt-ctl create` could detect and warn at
  net_allocate time rather than relying on the partial-repair path.

### Pre-Phase-1 oddity — devcontainer feature picks riscv64 over arm64

During `dc_build` the nanolayer feature resolver logged:

```
'shellcheck-v0.11.0.linux.riscv64.tar.xz' filtered by '\-ARM\-?|\-arm\-' 
'shellcheck-v0.11.0.linux.aarch64.tar.xz' filtered by '(?i)(arch)' (negative)
resolved asset: shellcheck-v0.11.0.linux.riscv64.tar.xz
```

The shellcheck install filter excludes `aarch64` (the correct arm
asset) and falls back to `riscv64`. On a non-aarch64-capable host
this would be a hard fail; on the Mac arm64 evidently the riscv64
binary doesn't survive past install. Not reproduced as a hard
failure during this run — flagging for the devcontainer-features
upstream owners.

### UX issues (one-liner each)

- `wt-ctl create` requires `branch` positional; conflicts with the
  prompt's "no `--branch` needed if invoked from a worktree on that
  branch" — `wt-ctl status` infers branch from cwd, `create` should
  too.
- `wt-ctl delete` and `wt-ctl create` have the `branch` positional
  un-documented in `--help` (no description text under
  `positional arguments`).
- `wt-ctl attach --help` shows only `positional arguments: args` with
  no description — users can't tell that no-args attaches via tmuxp
  and args exec verbatim. Same applies to `kfp`, which exposes
  `status/start/stop/register` only as a `{choices}` set with no
  per-subverb help body.
- The `evg spawn` error message used to drop the underlying rc/stderr;
  now fixed but worth a doc note that the message tells you what to
  do (set `MCK_DEVC_EVG_KEY_NAME` or pass `--key`).
- After a failed `evg_prepare`, `wt-ctl status` says
  `evg host    not-found`. That's correct given display_name=null on
  the freshly spawned host (because the rename portion of `host
  modify` failed). The status line is misleading — there IS an
  EVG host, just untagged. Consider falling back to "instance_tags
  contain wt-ctl=<name>" before reporting "not-found".
