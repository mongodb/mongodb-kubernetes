## Section C — Failures categorised

### Terminal blocker — host SSH-agent forwarding is not active

`onCreateCommand` runs `.devcontainer/scripts/on-create.sh`, which
chains to `.devcontainer/scripts/on-create/15-claude-plugins.sh`.
That script does:

```bash
claude plugin marketplace add git@github.com:10gen/core-platforms-ai-tools.git
```

Under `set -euo pipefail`. The SSH clone fails with
`Permission denied (publickey)` because the SSH agent inside the
devc has no keys loaded:

```
$ docker exec lsierant_kube-26-background-tester_devcontainer-devcontainer-1 \
    sudo -u vscode bash -c 'export SSH_AUTH_SOCK=/ssh-auth.sock; ssh-add -L'
Error connecting to agent: Permission denied
```

Root cause: `.devcontainer/compose.generated.yml` mounts
`/run/host-services/ssh-auth.sock:/ssh-auth.sock`, the standard
**Docker Desktop ssh-forwarding socket**. That path does not exist
on this host right now:

```
$ ls /run/host-services/ssh-auth.sock
ls: /run/host-services/ssh-auth.sock: No such file or directory
```

The host's actual ssh-agent is at
`/var/run/com.apple.launchd.UWTdZ8gna2/Listeners`
(macOS launchd) and a GPG-based agent at
`~/.gnupg/S.gpg-agent.ssh` — neither is what compose.generated.yml
binds. Docker creates the missing source as an empty placeholder
file rather than fail-fast, so the bind succeeds with a dud socket
and ssh sees an empty agent on the other end.

Two side-effects of the same root cause:

1. `on-create.sh` aborts at marketplace add → wt-ctl `dc_up`
   reports rc=1.
2. `evg-host-proxy` enters `Restarting (1)` loop with logs:
   ```
   evg-host-proxy: ssh-agent ready; starting autossh to ubuntu@ec2-54-155-4-40...
   ubuntu@ec2-54-155-4-40...: Permission denied (publickey).
   ```
   (autossh from inside the container hits the same dud agent.)

Phase-4 retry-1 (this morning, earlier) reached `post_start` —
that means the Docker Desktop ssh forwarder **was** populating
`/run/host-services/ssh-auth.sock` then. So this is a recurring,
flaky host-side state, not a code regression on
`lsierant/devcontainer`.

This is the **same class of issue** as the "DNS / proxy /
ghcr.io flake" the prompt warned about: host-side network/auth
state outside wt-ctl's control. Per the prompt's explicit guidance
(**"If DNS / proxy / ghcr.io flakes return: stop the run cleanly,
write a partial report, do NOT keep retrying"**), I stopped after
the first deterministic on-create failure rather than spin the
host-side debugger.

### Pre-existing — `on-create.sh` runs even when ssh-agent is dead

The `15-claude-plugins.sh` script does check for an SSH agent:

```bash
if [[ -z "${SSH_AUTH_SOCK:-}" ]] || [[ ! -S "${SSH_AUTH_SOCK}" ]]; then
    echo "SSH agent not forwarded; skipping claude marketplace setup."
    exit 0
fi
```

But this only validates that `SSH_AUTH_SOCK` points to **a socket
file**, not that the agent on the other side has any keys. The
empty placeholder Docker creates passes the `-S` test, so the
script proceeds and the marketplace clone is the first thing that
actually probes the agent. **UX recommendation**: probe via
`ssh-add -L >/dev/null 2>&1 || { echo "ssh-agent socket present
but unreachable; skipping marketplace setup."; exit 0; }`. That
would let dc_up succeed even when Docker Desktop's forwarder is
off — the user can pick up marketplace install at attach time.

Not fixed in this session — the change touches
`.devcontainer/scripts/on-create/`, which is shared
infrastructure across many worktrees and warrants explicit user
review before committing to `lsierant/devcontainer`.
