## Known limitations / follow-ups (do NOT block Phase 4/5)

1. **`evergreen host create` has no `--expiration` flag.** Hosts inherit the server default (≈24 h). `wt-ctl delete` mitigates the cost. Not worth a ticket; if it becomes painful, follow up with a `host modify --no-expire` + cron-stop combo or wait for the upstream CLI to gain `--expiration`.
2. **Host-id parsing from `host create` stdout is heuristic.** The fallback uses `host list --mine --json` and picks the freshest untagged host. Race window is single-developer-tiny but documented in the docstring. Phase 4/5 wet test will exercise this on a real spawn.
3. **SSH verify is warn-only.** If it fails, `spawn()` still returns the host id and lets `evg_prepare.sh` step 4 (`evg_host.sh ssh`) decide whether to abort. Aligns with `feedback_fail_fast_no_masking` on the orchestrator level while letting Phase 1 leave the orchestrator behavior unchanged.
4. **No live wet test inline.** Per the plan ("default to skipping live evergreen calls — let Phase 4/5 wet-test"). The commit body documents the wet-test commands for Phase 4/5 subagents to invoke implicitly via `wt-ctl create`.
5. **UX inconsistency (not blocking):** `wt-ctl network`, `wt-ctl om`, `wt-ctl ctx`, `wt-ctl wt`, `wt-ctl kfp` subparsers still register sub-verbs with no `help=` text. Phase 1 fixed only the `evg` family. Filing a small follow-up to backfill the rest would be a polish job, not a blocker.
