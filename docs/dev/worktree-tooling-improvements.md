# Worktree-dev tooling improvements — proposed fixes

Source: lessons from the KUBE-17 search-connectivity-tool autonomous
session (2026-05-09 → 05-10), where the agent hit eight non-trivial
gotchas in the `wt_setup.sh` / `dc_up.sh` / `e2e_run.sh` / `op_run.sh`
flow that all currently require workarounds. The workarounds are now
documented in the `worktree-dev` skill (commit `1b32dae` on
`lsierant/mck-dev-rules`); this plan proposes the source-side fixes
that would make the workarounds unnecessary.

Filed under `docs/dev/` rather than as a JIRA epic because, per the
standing rule, devcontainer/orchestrator work lands directly on
`lsierant/devcontainer` without JIRA tickets — this doc is the
backlog.

## #1 — `op_run.sh` shouldn't inherit host-side `PATH`

**Problem.** `op_run.sh` starts the operator in a tmux session via
`bash -lc 'set -a; source .generated/context.export.env; source
.generated/context.operator.export.env; set +a; ...; go run ./main.go'`.
When `context.export.env` was generated on the host (typical: the
worktree's first `make switch` runs on macOS before the devcontainer
exists), it carries a macOS `PATH` like
`/Users/lukasz.sierant/.local/bin:...:/opt/homebrew/...:/usr/local/go/bin:...`.
Sourcing that inside the container clobbers the container's PATH
(`/usr/local/go/bin:/go/bin:...`) with macOS paths that don't exist
in the container — `go: command not found`.

The container's login shell (via `bash -lc`) DOES run `/etc/profile.d/`
which sets `PATH` correctly, but the subsequent `source
context.export.env` overwrites it.

**Root cause.** `context.export.env` exports the *running shell's*
`PATH` rather than computing it for the target side. The file is
generated by `scripts/dev/contexts/root-context` (which `set_env_context.sh`
sources). `root-context` does `export PATH="${PATH}"` based on the
host shell's PATH at generation time.

**Proposed fix — option A (preferred):** `op_run.sh` should NOT source
`context.export.env` wholesale. Instead, source only the
operator-needed variables (`KUBECONFIG`, `OPERATOR_ENV`,
`WATCH_NAMESPACE`, `MONGODB_REPO_URL`, `OPS_MANAGER_*`, …) and leave
`PATH` alone:

```bash
# In op_run.sh, replace the wholesale source with:
set -a
source <(awk '/^export (KUBECONFIG|OPERATOR_ENV|WATCH_NAMESPACE|NAMESPACE|MONGODB_|OPS_MANAGER_|VERSION_UPGRADE_|READINESS_PROBE_)/' \
    .generated/context.export.env .generated/context.operator.export.env)
set +a
```

This keeps the container's PATH (set by `bash -lc`'s profile.d) intact
while still picking up the operator's runtime env.

**Proposed fix — option B (alternative):** `root-context` should NOT
write `PATH` at all. Callers that need a particular toolchain in
PATH should set it themselves (the container does, the host does).
This is cleaner architecturally but requires auditing every caller of
`set_env_context.sh` to confirm none of them rely on the exported
PATH (the EVG host setup path notably does — see
`scripts/dev/setup_evg_host.sh`).

**Recommendation:** option A — surgical, no callers beyond `op_run.sh`
need touching. Option B is the right end-state but a separate piece
of work.

**Verification.** After the fix:
```bash
devcontainer exec --workspace-folder $WT bash -lc '
  source .generated/context.export.env
  source .generated/context.operator.export.env
  which go && go version
'
# Expect: /usr/local/go/bin/go + go1.25.x linux/...
```

## #2 — `set_env_context.sh` should fail-fast (not just warn) on side-mismatch

**Problem.** When `context.export.env` was generated on side `host`
and is being sourced on side `container` (or vice versa),
`set_env_context.sh` prints:

```
(!!) /workspace/.generated/context.env was generated on side='host'
but you're sourcing it on side='container'. Re-run 'make switch'
(or scripts/dev/switch_context.sh <ctx>) on the container side to
regenerate it.
```

…and then continues. Anything sourced from the file *still* applies,
including the broken PATH. Errors that follow (e.g. `go: command
not found`) look unrelated to the warning and you have to grep
backward through the log to find it.

**Root cause.** The check is an `echo` not an `exit 1`.

**Proposed fix.** Make the side-mismatch a hard failure when sourced
in a script that doesn't pass an opt-out (e.g. `MCK_ALLOW_STALE_CONTEXT=1`).
Interactive shells can keep the warning-and-continue behaviour for
diagnostics. Concretely in `set_env_context.sh`:

```bash
if [[ "${file_side}" != "${current_side}" ]]; then
  echo "(!!) context.env was generated on side='${file_side}' but you're sourcing it on side='${current_side}'." >&2
  echo "Re-run 'make switch' on the ${current_side} side." >&2
  if [[ "${MCK_ALLOW_STALE_CONTEXT:-0}" != "1" ]]; then
    echo "Aborting. Set MCK_ALLOW_STALE_CONTEXT=1 to override." >&2
    return 1  # or `exit 1` if invoked directly
  fi
fi
```

The orchestrator calls it from `wt_setup.sh` after every `make
switch`, so the abort path should never trigger in the happy case;
it only triggers in the partial-recovery case where the user thinks
they switched but actually didn't.

**Verification.** Force the broken state and confirm the abort:
```bash
# Generate on host, then source in container
make switch context=e2e_mdb_kind_ubi_cloudqa     # on host
devcontainer exec --workspace-folder $WT bash -lc '
  source /workspace/.generated/context.export.env
'
# Expect: non-zero exit, error message, no PATH pollution.
```

## #3 — `k8s-proxy` should reload its kubeconfig on start

**Problem.** `k8s-proxy` is configured at runtime via
`curl -X PATCH .../kubeconfig`. The proxy holds the kubeconfig in
memory only — any container restart wipes it (logs show `client
manager rebuilt {contexts: 0}`), and subsequent `*.svc.cluster.local`
lookups silently NXDOMAIN. Symptom from the e2e side: `pymongo`
client raises `[Errno -2] Name or service not known`. Pods are
healthy, kubeconfig is valid on disk, but the proxy doesn't know.

**Root cause.** `k8s-proxy` (binary from
`https://github.com/mongodb/kube-forwarding-proxy`) only accepts the
kubeconfig via the runtime HTTP API. There's no on-disk-load-on-start
path.

**Existing partial fix.** `lsierant/devcontainer@77e708164` re-pushes
the kubeconfig from `.devcontainer/scripts/post-start.sh` on every
container post-start. This covers the standard `dc_up` cycle but not
manual `docker restart <project>_devcontainer-k8s-proxy-1`.

**Proposed fix — option A (preferred, cross-repo):** add a
`--kubeconfig <path>` flag to `k8s-proxy` that loads the file at
startup. Mount `.generated/evg-host.devc.kubeconfig` into the
container at a known path (`/etc/k8s-proxy/kubeconfig`) via
`compose.yml`. Set the flag in the entrypoint. Runtime PATCH still
works for re-registration, but startup is self-sufficient.

**Proposed fix — option B (workaround in this repo):** use a
sidecar / init container that polls the proxy's `/healthz` (or
`/contexts`) endpoint after start, and re-PATCHes the kubeconfig
when it sees `contexts: 0`. Compose-only; no upstream change. Less
elegant but landable today.

**Recommendation:** option B as a stopgap, item #10 in the tracker
captures it. Option A is the right end-state and should be filed
upstream against `mongodb/kube-forwarding-proxy`.

**Verification.** Restart `k8s-proxy` and confirm DNS still works:
```bash
docker restart <project>_devcontainer-k8s-proxy-1
sleep 5
devcontainer exec --workspace-folder $WT bash -c '
  getent hosts kubernetes.default.svc.cluster.local
'
# Expect: 172.x.y.z (a VIP), not empty.
```

## #4 — Tunnel-probe should retry kubeconfig re-push

**Problem.** The tunnel-probe in `wt_setup.sh` (around line 308) loops
`kubectl get nodes` for 60s. If the autossh tunnel comes up but
`k8s-proxy` lost its kubeconfig (because the container restarted
inside the 60s window), `kubectl` keeps failing even though the
SOCKS5 path is healthy — the proxy returns NXDOMAIN. The probe
times out and the script aborts.

**Root cause.** The probe only retries one thing (kubectl reachability)
when the failure can be in two places (tunnel OR proxy state).

**Proposed fix.** Inside the probe loop, on each failed `kubectl get
nodes`, also re-push the kubeconfig:

```bash
while ! kubectl get nodes >/dev/null 2>&1; do
  if [[ $(date +%s) -ge ${deadline} ]]; then
    echo "ERROR: kubectl tunnel not ready within 60s." >&2
    exit 1
  fi
  # Defensive: if k8s-proxy lost state during a restart, re-push.
  curl -fsS -X PATCH --data-binary @/workspace/.generated/evg-host.devc.kubeconfig \
    "http://${K8S_FWD_PROXY}/kubeconfig" >/dev/null 2>&1 || true
  sleep 2
done
```

Cheap, idempotent, no false positives.

## #5 — Tunnel-probe should warn-and-continue, not abort

**Problem.** When the probe fails, `wt_setup.sh` exits before running
`make prepare-local-e2e`. The cluster comes up, the namespace
doesn't have CRDs, and the next e2e run fails with `namespaces
"ls-25" not found`. Recovery is documented in the skill but the
right behaviour is "don't drop the rest of the pipeline on the
floor".

**Root cause.** Probe failure is treated as fatal. It's actually
diagnostic — we want a clear error message at the right place
(rather than a misleading error five layers deep), but downstream
phases can still attempt their work. If the cluster truly is
unreachable, `make prepare-local-e2e` will fail loudly anyway.

**Proposed fix.** Convert the probe from "abort" to "warn":

```bash
if ! probe_kubectl_tunnel; then
  echo "WARN: kubectl tunnel not ready within 60s. Continuing — prepare-local-e2e will surface a clearer error if the cluster is genuinely unreachable." >&2
fi
```

Combined with #4's retry-with-rekubeconfig, the probe will rarely
fail in practice anyway.

**Risk.** Slightly worse error message in the genuinely-broken case
(cluster down, EVG host hung). Mitigation: keep a clear `WARN: ...`
at the top of the surviving phases' logs.

## #6 — `e2e_run.sh` path-arg resolution

**Problem.** The script has:

```bash
cd docker/mongodb-kubernetes-tests
exec pytest "../../${target}"
```

`target=tests/search/foo.py` resolves to `/workspace/tests/search/foo.py` —
non-existent. The actual location is
`/workspace/docker/mongodb-kubernetes-tests/tests/search/foo.py`.

**Root cause.** The script's example header says `e2e_run.sh
tests/search/foo.py` but the implementation expects `e2e_run.sh
docker/mongodb-kubernetes-tests/tests/search/foo.py`. Mismatch
between docs and code.

**Proposed fix.** Normalise the path before invoking pytest:

```bash
# If the caller passed a tests/-rooted path, prepend the
# docker/... prefix automatically.
if [[ "${target}" == tests/* ]]; then
  target="docker/mongodb-kubernetes-tests/${target}"
fi

cd /workspace
exec pytest "${target}"
```

Also: `cd /workspace` instead of `cd docker/mongodb-kubernetes-tests`
keeps the path machinery sane and matches the existing absolute-path
case (which already uses `/workspace/...`).

**Verification.**
```bash
e2e_run.sh tests/search/search_connectivity_tool.py
e2e_run.sh docker/mongodb-kubernetes-tests/tests/search/search_connectivity_tool.py
e2e_run.sh e2e_search_connectivity_tool   # marker form, unchanged
# All three should hit the same test set.
```

## #7 — `make full-reset` target

**Problem.** `make prepare-local-e2e` is named like a "set up
everything" but in reality it does targeted preparation: namespace
ensure, OM project delete, configmap/secret create. It does NOT
delete the namespace, bounce stuck pods, or remove CRDs. When the
cluster is in a wedged state (mongod agents on a deleted OM project),
prepare-local-e2e doesn't fix it — you have to manually delete pods.

**Root cause.** No target exists for "wipe everything in the test
namespace and re-create it from scratch".

**Proposed fix.** New `scripts/dev/full_reset.sh` + `make full-reset`
target:

```bash
#!/usr/bin/env bash
# full_reset.sh — true clean-slate of the test namespace.
# 1. Delete all MongoDB CRs (graceful)
# 2. Delete the namespace
# 3. Wait for namespace deletion
# 4. Delete the OM project (delete_om_projects.sh)
# 5. Re-run prepare-local-e2e to recreate everything fresh

set -Eeuo pipefail
source scripts/dev/set_env_context.sh

echo "==> Deleting all MongoDB CRs in ${NAMESPACE}"
kubectl delete mdb,mdbu,mdbs,mdbms,om -n "${NAMESPACE}" --all --timeout=120s || true

echo "==> Deleting namespace ${NAMESPACE}"
kubectl delete namespace "${NAMESPACE}" --timeout=180s || true

echo "==> Waiting for namespace ${NAMESPACE} to be fully deleted"
while kubectl get namespace "${NAMESPACE}" >/dev/null 2>&1; do
  sleep 2
done

echo "==> Re-running prepare-local-e2e"
exec make prepare-local-e2e
```

```makefile
full-reset: ## Wipe and re-create the test namespace + OM project
	scripts/dev/full_reset.sh
```

**Risk.** The `kubectl delete namespace` is destructive — anything
the user manually placed in the namespace (e.g. test data, debugging
configmaps) is gone. Document this clearly in the help text.

## #8 — `evg_patch_search.sh` wrapper

**Problem.** The canonical EVG-patch invocation for a search-relevant
PR is:

```bash
evergreen patch -p mongodb-kubernetes \
  --variants unit_tests,e2e_mdb_kind_ubi_cloudqa_large,e2e_static_mdb_kind_ubi_cloudqa_large \
  --regex_tasks 'lint_repo|unit_tests|<MARKER>' \
  --finalize \
  --yes \
  --description '<KUBE-N> — pre-commit + unit + lint + <MARKER>'
```

…which nobody remembers verbatim. `--tasks all` (the wrong default)
fans out to 20+ unrelated tasks and burns EVG capacity.

**Proposed fix.** New `scripts/dev/evg_patch_search.sh <marker>
<description>`:

```bash
#!/usr/bin/env bash
set -Eeuo pipefail
test $# -ge 2 || { echo "Usage: $0 <marker> <description>"; exit 1; }
marker="$1"; shift
description="$*"
exec evergreen patch -p mongodb-kubernetes \
  --variants unit_tests,e2e_mdb_kind_ubi_cloudqa_large,e2e_static_mdb_kind_ubi_cloudqa_large \
  --regex_tasks "lint_repo|unit_tests|${marker}" \
  --finalize --yes \
  --description "${description}"
```

Use:
```bash
scripts/dev/evg_patch_search.sh e2e_search_connectivity_tool 'KUBE-17 — search connectivity tool'
```

## #9 — `wt_teardown.sh --purge-orphan-namespaces`

**Problem.** Namespace rotates per-run (`ls-25` → `ls-27` etc.).
Stale resources from broken runs stay in the cluster until the
worktree is torn down. Cumulative across many runs in a single
worktree session, the kind cluster fills up with dead StatefulSets,
PVCs, and OM-related secrets.

**Root cause.** `wt_teardown.sh` only deletes the *current*
namespace; older rotations are orphaned.

**Proposed fix.** Optional flag:

```bash
wt_teardown.sh --purge-orphan-namespaces <branch>
```

When set, list all namespaces matching the prefix used by the
worktree (e.g. `ls-*`) and delete them. Skip the current one (which
gets cleaned up by the regular teardown anyway).

**Risk.** Cross-worktree pollution if two parallel worktrees use the
same namespace prefix. The KUBE-70 fix already suffixes namespaces
with `MCK_DEVC_NET_PREFIX`, so each worktree's namespaces are
distinguishable. Document this assumption.

## #10 — `dc_up.sh` post-start re-push only on empty contexts

**Problem.** The current `post-start.sh` re-pushes the kubeconfig
unconditionally on every container start. For a healthy stack
that's a wasted round-trip; for a stack where `k8s-proxy` already
has the kubeconfig loaded and is mid-request, it's a brief
disruption.

**Root cause.** No check is performed before re-pushing.

**Proposed fix.** Query `k8s-proxy`'s `/contexts` endpoint first; if
non-empty, skip the re-push:

```bash
contexts=$(curl -fsS "http://${K8S_FWD_PROXY}/contexts" 2>/dev/null | jq -r '.contexts | length' 2>/dev/null || echo 0)
if [[ "${contexts}" == "0" ]]; then
  echo "post-start: k8s-proxy has 0 contexts; re-pushing kubeconfig"
  curl -X PATCH --data-binary @/workspace/.generated/evg-host.devc.kubeconfig \
    "http://${K8S_FWD_PROXY}/kubeconfig"
fi
```

Cheap, idempotent, no behaviour change in the recovery path.

**Risk.** If `k8s-proxy`'s `/contexts` endpoint reports a stale
context that's no longer reachable (e.g. EVG host changed IP), the
re-push gets skipped. Mitigation: check for both empty AND
unreachable; or just keep the unconditional re-push and call this
fix not-worth-it.

  guardrail; the merge-from-parent workaround documented in the
  skill is the intended escape hatch.

  fix (already in the skill); not a tooling change.

  `evergreen-ci/evergreen`. Not blocking us.
